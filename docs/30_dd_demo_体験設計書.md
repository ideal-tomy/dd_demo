# DDデモ 体験設計書 — フェーズ×AI体験マップ
対象: 自己資金で買収→自社で再生→売却まで一気通貫で行うプリンシパル型企業
目的: 「診断が出る」デモから「作業が消え、人間が思考に戻る」デモへの転換

---

## 0. 設計思想:感動の方程式

現状のデモに足りないのは機能ではなく**Before(作業の重さ)の可視化**。「7.2億円です」と結果だけ見せても、それを人手でやると何が起きるかを知らない画面では価値が伝わらない。感動は次の三段で作る:

> **①Beforeを見せる**(この作業、通常は担当2名で3日) → **②目の前で消える**(AIが数十秒で実行、過程も見える) → **③浮いた時間の使い道を返す**(「ここから先はあなたの経営判断です」と問いを渡す)

③が本デモの思想の核。AIの終着点を「答え」ではなく**「人間が考えるべき問い」**にすることで、「AIが人を置き換える」ではなく「AIが人を思考に戻す」というメッセージがUI自体から伝わる。

### 0-1. 誠実さの原則(演出と能力の境界)

デモに求められる誠実さは「全部が本物であること」ではない。**「どこが本物でどこが演出かを、聞かれた瞬間に自分が即答できること」**である。プロはデモが製品でないことは百も承知で、嘘だけを嫌う。

| 層 | 許否 | 例 |
|---|---|---|
| **見た目と速度**(演出してよい) | 可 | ライブ実況のストリーム表示、ロールアップ、差分フラッシュ、体感速度の演出 |
| **能力**(演出してはいけない) | 不可 | 決定的な事前結果を「いま解析しました」と能力偽装すること |

崩壊するのは「見せ方」ではなく、**能力の偽装**だけ。したがって「デモか本物か」の二択は偽の問いである。答えは**二層**にする:

> **シナリオ層(明示した安定部分) + ライブ層(1本だけの本物)**

#### シナリオ層(サンプル企業モード)
- 既存どおり決定的計算・テンプレ findings でよい
- 画面に小さく明示する: **「サンプル企業は事前診断済みのシナリオです」**
- この一行のコストはゼロ。隠すより信頼が上がり、「あくまでデモ」の気持ちにも合致する

#### ライブ層(実解析1本)
- **給与台帳CSV × 勤怠の突合だけ**を本物の解析処理にする
- 同梱はミナトテック方式の架空データ(自作)。結果は既知なので商談が事故らない
- しかし解析処理そのものは本物なので「御社のデータでも同じことが起きます」と言い切れる
- 「体験を」の気持ちはこの1本が満たす。ライブ実況は見せ方として演出してよい(結果・根拠行は実解析由来)

### 0-2. 商談タイプ別・必須セット

実装・デモ準備のぶれを防ぐため、商談タイプごとに見せる最小セットを固定する。

| タイプ | 相手の関心 | 必須セット |
|---|---|---|
| **再生・導入寄り** | PMI・自社実行・ツール導入 | タイムカウンター + 「あなたへの問い」 + **P4レバー→実デモ接続** + シナリオ層の明示 |
| **DD能力寄り** | 資料読解・発見の本物感 | タイムカウンター + 「あなたへの問い」 + **ライブ層(給与×勤怠の実解析1本)** + シナリオ層の明示 |

どちらでも共通: 全面リデザインは必須ではない(§4の段階導入)。

## 1. 感動を作る5原則(全機能共通)

1. **Beforeの明示**: 各機能カードに「従来: ○時間/○人日」を小さく常設。実行後に「約○時間分の作業を代行」と表示し、セッション累計を右上に積算(タイムカウンター)。カウンターは**実務工数の試算**であり、内訳で根拠を即答できること(§3-1)
2. **根拠に触れられる**: すべての発見は「根拠を見る」で原文該当箇所(台帳の行・契約の条文)へジャンプ。プロは根拠のない指摘を信用しない。ライブ層では根拠行は実解析由来であること
3. **成果物が仕事の形**: 出力はExcel(Q&Aシート)・Word(報告書ドラフト)・PDFでダウンロード可能。「画面で見る」で終わらせない
4. **人間の判断ポイントの明示**: 各フェーズ出力の最後に「判断が必要な問い」を2〜3個返す。問いは**AIに毎回考えさせず、企業JSONの`questions`(フェーズ×主軸別)在庫から出す**——レバーと同じ思想。儀式化を防ぐため、フェーズ固有の判断軸に縛る
5. **過程が見える**: 解析中は無言のスピナーではなくライブ実況をストリーム表示。実況は見せ方(演出可)。能力の証明はライブ層の実解析結果と根拠ジャンプに置く

## 2. フェーズ×AI体験マップ

### Phase 0:ソーシング・初期検討
- **実務の痛み**: ノンネーム・IMが大量に届き、見るべき案件の選別に経験者の時間が溶ける
- **AI体験**: 匿名指標(業種・売上・従業員数・依存度)を入れると業種priorから「この型の簿外はここに出る」を即座に提示(既存「自社で入力」の再定義)
- **感動ポイント**: 「データを見る前から当たりがつく」— 機密情報ゼロで動く点を明示し、機密性プレゼンと接続
- **成果物**: 初期検討メモ(1枚)
- **実装状態**: ✅ほぼ済み(位置づけの言語化のみ)

### Phase 1:DD準備 — 資料が読まれる【ライブ層の置き場】
- **実務の痛み**: データルームの数百ファイルを読み、突合する。DD工数の最大部分(数人×数週間)
- **AI体験**:
  - **シナリオ層**: サンプル企業選択時は決定的 findings。注記「サンプル企業は事前診断済みのシナリオです」を常設
  - **ライブ層**: 同梱の給与台帳CSV+勤怠を「データルームに投入」→(実況は演出可)→**実解析**で「未払残業の兆候: ○月の打刻と支給額の乖離(根拠: 台帳○行目)」
- **一気通貫の予告(最安)**: 発見事項カードに1行タグ  
  `EXIT時: 処置済みとして開示可 → 評価ディスカウント回避`  
  DD発見が売る時のクリーンさの証明に反転することを、P1の時点で予告する(P5実装を待たない)
- **感動ポイント**: ライブ層で「解析処理は本物・データは架空だが既知」と言い切れること。シナリオ層は隠さず明示する
- **成果物**: 発見事項リスト(根拠リンク付き)+EXIT予告タグ
- **実装状態**: 🔴新規(ライブ層1本が最重要。シナリオ層は既存+明示)

### Phase 2:DD実行 — 質問と報告が書かれる
- **実務の痛み**: 対象会社へのQ&Aリスト(質問状)と追加依頼資料リストの作成。抜け漏れが怖く、毎回ゼロから書く
- **AI体験**: 発見事項から質問リスト(カテゴリ・優先度・依頼資料付き)を自動生成→Excelダウンロード。さらに中間報告書ドラフト(Word)
- **感動ポイント**: 成果物が「いつも作っている納品物の形」で出てくる=明日から使えると確信する瞬間
- **成果物**: Q&Aシート(xlsx)/DD中間報告書ドラフト(docx)
- **実装状態**: 🔴新規(DD能力寄りの商談で前面。再生寄りでは後回し可)

### Phase 3:買収実行 — 論点が契約に変換される
- **実務の痛み**: 発見事項をSPA(株式譲渡契約)の建付け(価格調整/エスクロー/表明保証/CP)に落とす整理
- **AI体験**: 既存のoffbalance_treatmentを発展させ、簿外項目ごとに「処理方法の選択肢と金額影響」をSPA論点リストとして出力
- **問い**: `questions`在庫から、例として「価格調整で処理するか、是正投資して高値EXITを狙うかは、保有期間の思想次第です」等を主軸別に出す
- **感動ポイント**: 主軸タブで処置方針が変わる既存演出が、そのまま契約戦略の比較になる
- **成果物**: SPA論点整理表
- **実装状態**: 🟡改修(既存機能の再編成)

### Phase 4:PMI・再生実行 — レバーが実物になる【再生・導入寄りの主戦場】
- **実務の痛み**: 買収後のテコ入れを自分たちの手で回す。100日プラン、KPI定着、現場へのシステム導入
- **AI体験**: **バリューアップレバーのカードに「このツールを試す→」ボタンを付け、実際に動く既存デモが開く**(AI配車→配車デモ/ナレッジ共有→ミナトテック工場ボット/日報デジタル化→OCRデモ/シフト最適化→シフトデモ)
- **戻り導線(必須)**: ツールデモ側に「DD画面に戻る」。**戻り後は元のレバーカードが展開したまま・スクロール位置を保持**(商談の流れが物理的に止まらないこと)。クエリで案件・主軸・レバーIDを保持
- **感動ポイント**: 試算上のレバーが絵空事ではなく、今ここで触れる実物として存在する。「導入するツールがもう動いている」— 自ら手を動かす会社への最大の説得力。かつAXEON/idealにとって買収先への導入受注という商流が開く
- **成果物**: 100日プラン(レバー別・KPI付き)
- **実装状態**: 🟡改修(リンク接続+状態保持)+既存デモ資産の流用

### Phase 5:EXIT — 売り物として磨かれる
- **実務の痛み**: データルーム整備、買い手別の訴求整理、IM作成
- **AI体験**: 処置済み/未処置の簿外一覧からデータルーム整備チェックリストを生成。買い手類型別(3PL/同業/ファンド)にIMの訴求章立てをドラフト
- **感動ポイント**: DD時代の発見事項が、売る時の「クリーンさの証明」に反転する一気通貫の物語(初回はP1のEXIT予告タグ+口頭でカバー可)
- **成果物**: 整備チェックリスト/IM章立てドラフト
- **実装状態**: 🔴新規(優先度低・語り+P1タグでカバー可)

### 横断:案件が資産になる
- **AI体験**: ①ポートフォリオビュー(保有各社のEBITDAブリッジ進捗を横断表示) ②house view登録(自社チェックリストを業種priorに追加すると次案件から自動で観点に入る — 5スロットスキーマ思想の流用)
- **感動ポイント**: 「使うほど御社の目利きがシステムに蓄積される」。中抜き構造への対抗というミッションと響き合う
- **実装状態**: 🔴新規(構想として商談で語る)

## 3. 「未知の体験」を作る3つの演出

演出してよいのは見た目と速度。能力は偽装しない(§0-1)。

### 3-1. タイムカウンター(シグネチャー)
- 画面右上に「このセッションでAIが代行した作業: 約14.5時間」を積算表示。**加算は各機能の完了時のみ**
- 数字の恣意性そのものは問題ではない。問題は**根拠を即答できないこと**
- 内訳ポップに必ず書く:
  - 機能別の代行時間
  - **算定根拠: 標準的なDD実務工数(○○を参照)に基づく実務時間換算の試算**
- 「試算です」と堂々と言える数字にする。内訳を開いたときに根拠が出ること自体が「作り込んだ」証明になる
- デモ終了時、この数字が締めの一言と連動する

### 3-2. 解析ライブ実況
- ストリーミングで「就業規則 第15条を確認… デジタコ記録と照合… 3月に乖離検出」と過程を流す
- **見せ方としての演出は可**。能力の証明はライブ層の実解析結果と根拠行に置く
- シナリオ層では「事前診断済み」注記があるため、実況を付けても能力偽装にならない(聞かれたら即答: シナリオは決定的、ライブ層だけ実解析)

### 3-3. 「あなたへの問い」カード
- 各フェーズ出力の最後に固定配置。見出し思想:「この3つはAIには決められません」
- 問いは企業JSON `questions[phase][strategy_axis]` の在庫から2〜3個。フェーズ固有の判断軸に縛り、使い回しを避ける
  - 例 P1/P3: 価格調整 vs 是正投資(保有思想)
  - 例 P4: 100日で先に切るレバーはどれか
- AIに毎回生成させない(レバー在庫と同じ思想)

## 4. 改修の優先順位(段階導入)

全面リデザインは受注後でよい。**カウンター+問いカードを先に載せて商談に出す**。

| 優先 | 施策 | 工数感 | 期待効果 |
|---|---|---|---|
| **先行(商談投入)** | タイムカウンター(根拠付き内訳)+「あなたへの問い」カード(`questions`在庫) | 小 | 思想をUIで語れる最小セット |
| **先行** | シナリオ層の明示注記「事前診断済みのシナリオです」 | 極小 | 信頼。コストゼロ |
| **P1・再生寄り** | レバー→既存デモ接続+**戻り時の展開/スクロール状態保持**(Phase 4) | 小 | 相手の主戦場に直撃 |
| **P1・DD寄り** | ライブ層: 給与CSV×勤怠の実解析1本(+実況は演出可) | 中〜大 | 「能力は本物」の証明。事故らない架空データ |
| P2 | 発見事項のEXIT予告タグ1行 | 極小 | 一気通貫の物語を最安で予告 |
| P2 | Q&Aシート生成+Excelダウンロード(Phase 2) | 中 | 「明日から使える」の決定打 |
| P3 | SPA論点リスト再編成 | 小 | 既存機能の価値言語化 |
| P3 | EXIT整備チェックリスト/ポートフォリオ/house view | 大 | 初回は口頭+P1タグで十分 |
| 後続 | 全面UIリデザイン(委員会モード・二分割ワークベンチ等) | 大 | **受注後**。先行商談では必須としない |

## 5. 商談の締めの一言(案)

「AIがやったのは読むことと書くことです。**どの会社を買い、どこまで直し、いつ売るか——考える仕事は全部、御社に残っています。**そのための時間を返すのがこのツールです。」

必要なら即答できる補足:
- 「サンプル企業の数値はシナリオです。いま動かした給与×勤怠の突合だけが、御社データでも同じエンジンが走ります。」
- 「返した時間は標準DD工数に基づく試算です。内訳に根拠を出しています。」
